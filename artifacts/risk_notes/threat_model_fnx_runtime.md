# Threat Model — fnx-runtime

## Subsystem Scope
Strict/hardened mode dispatch, decision-theoretic policy engine, evidence ledger, compatibility gates.

## Trust Boundary
Internal — policy decisions based on parser-reported evidence signals. The runtime itself does not parse external input.

## Threat Categories

### 1. Mode confusion / downgrade attack
- **Attack:** Code path that checks mode in one place and assumes it in another; or a code path that falls through to Allow when it should FailClosed
- **Impact:** hardened-mode safety guarantees silently lost
- **Current mitigation:** `CgsePolicyEngine::evaluate_at` is the single decision point; all mode-sensitive behavior routes through it
- **Residual risk:** future code may add mode checks outside the engine → divergence

### 2. Decision ledger tampering
- **Attack:** Modifying `decision_ledger.jsonl` after the fact to hide a FailClosed event
- **Impact:** audit trail integrity lost
- **Current mitigation:** RaptorQ sidecar on the ledger (D7 planned); blake3 source_hash in envelope
- **Residual risk:** ledger is append-only in normal operation but the file is writable; no cryptographic chaining yet

### 3. Evidence signal manipulation
- **Attack:** Parser emits misleading evidence signals (e.g., reports low incompatibility_probability on a genuinely malformed input)
- **Impact:** Bayesian admission controller makes wrong decision
- **Current mitigation:** evidence signals are generated by the same parser that reports the issue → no trust boundary between them
- **Residual risk:** this is inherently a self-attesting system; external validation of evidence quality requires the conformance corpus

## Recommended Actions
1. Add mode-exhaustiveness tests: for every FailClosed trigger, verify that the same input produces Allow or FullValidate in strict mode
2. Add ledger append-only invariant test (verify line count only grows, never shrinks)
3. Consider cryptographic chaining (hash chain) for the decision ledger
