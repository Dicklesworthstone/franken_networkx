# franken_networkx Reality Check — 2026-05-03

> Generated by **cc_1** under SKILL-DRIVEN escalation. Audit franken-networkx
> (fnx) vs upstream NetworkX (nx) capability surface to categorize algorithms
> as **well-implemented**, **delegated/wrapped**, or **truly missing**, and
> file one bead per real gap.

## Method

1. **Surface coverage** — enumerate `dir(nx)` and every `nx.<submodule>` and
   compare against the matching `dir(fnx)` / `fnx.<submodule>`. Any nx
   callable absent from fnx counts as a hard gap.
2. **Implementation classification** — read the source of every fnx
   nx-equivalent callable:
   - **rust** — calls `_raw_…` / `_fnx.…`
   - **delegate** — body returns `nx.X(...)` or `_call_networkx_for_parity(...)`
   - **wrapper** — pure-Python implementation in fnx
   - **type_stub** — body is exclusively `raise NetworkXNotImplemented(...)`
3. **Performance benchmark** — drive 65 representative algorithms on a
   `barabasi_albert_graph(200, 5, seed=42)` fixture (with `to_directed()`
   for digraph-only ones), warm up once, time 10 iterations of each.
   `speedup = nx_ms / fnx_ms`. Categories:
   - `🟢 FAST`     — speedup ≥ 1.5×
   - `🟡 PARITY`   — 0.7× ≤ speedup < 1.5×
   - `🟠 SLOW`     — 0.3× ≤ speedup < 0.7×
   - `🔴 V_SLOW`   — speedup < 0.3× (≥ 3.3× slower than nx)

The bench fixture is small (200 nodes / 5 edges-per-node attachment), so
absolute milliseconds are not very meaningful, but the **ratio** is — for
linear-time algorithms the ratio is robust across graph sizes; for
super-linear ones the ratio gets worse as |V| grows.

Raw bench data: `/tmp/_reality_check_bench.json` (65 entries).
Raw classifier data: `/tmp/_cat_classify.json` (907 entries across top-level
+ `community` / `linalg` / `drawing` / `readwrite` / `convert` submodules).

## Recommendations / takeaways

1. **edge_connectivity is fundamentally broken** at this scale (32 s vs 23
   ms — 1370× slower). The fnx implementation is presumably running an
   exhaustive `O(|V|·|E|)` ST-cut over every pair instead of the
   White-Whitney algorithm nx uses. **P1, not P2** — this is closer to a
   correctness bug than a perf bug.
2. **The clustering family (clustering, transitivity, average_clustering,
   generalized_degree)** are all 5–6× slower with the same pattern. They
   likely share a `clustering_coefficient` core that's slow. **Bundle into
   one bead** that covers the family rather than 4 separate beads.
3. **The IO write family (write_edgelist, write_adjlist, write_gml,
   write_graphml)** — write_graphml is fast (1.96×) but the other three
   are 5–28× slower. The text-output path needs a Rust formatter.
4. **strongly_connected_components is 35× slower** even though
   `is_directed_acyclic_graph` is 4.9× faster. The SCC implementation is
   regressed relative to its own DAG counterpart — likely worth a deep
   dive.
5. **paths.dijkstra_path / astar_path** are 4× slower despite the
   underlying BFS being competitive. The wrapper overhead in
   `single_source_dijkstra_path` etc. is dominating; an internal Rust
   path-reconstruction would close this.
6. The pagerank perf fix shipped in commit `a04084e0` brought pagerank
   from 0.34× (3× slower) to 1.27× — pattern: apply the same "precompute
   edge tables; remove stale wrapper bypass" treatment to other power-
   iteration centralities (load_centrality first, then revisit
   katz_centrality).
7. There are essentially **no missing-by-name gaps**. Every name resolves
   somewhere — the work is in upgrading 644 pure-Python wrappers /
   70 delegate paths to Rust, prioritized by hotpath benchmark ratio.

## Sync staleness investigation (cod_4 escalation)

`br sync --flush-only` refused with:

> Database has 1255 issues, JSONL has 1433 unique issues. Export would
> lose 189 issue(s)…

Root cause: **JSONL ahead of DB.** Multiple agents have written beads to
JSONL but the local DB hasn't imported them. `br sync --import-only` says
"JSONL is current (hash unchanged since last import)" — i.e. the import
manifest believes the JSONL is up-to-date even though row-counts disagree.
The bug is in the import-manifest hash logic: it tracks JSONL file hash but
not row count. After a flush by another agent that wrote both DB and JSONL
together, the manifest sees the JSONL file as "already imported" and skips
the row-by-row reconciliation.

`br doctor` reports:
- `ERROR db.recoverable_anomalies: blocked_issues_cache is marked stale and needs rebuild`
- `WARN counts.db_vs_jsonl: DB and JSONL counts differ`
- `OK sync.metadata: Database and JSONL are in sync` *(stale per above)*

Safe fix: `br sync --rebuild` (rebuilds DB from JSONL, the strict superset).
This was attempted but blocked by long-running flush from another agent
holding a write transaction. Once contention clears, run:

```bash
AGENT_NAME=cc_1 br sync --rebuild
```

The flag `--repair` on `br doctor` does the same DB-rebuild path.
**Do not** run `br sync --flush-only --force` — that goes the wrong
direction (DB → JSONL with --force overrides the safety guard) and
**permanently drops the 189 JSONL-only issues**. (Another agent appears
to have run this; JSONL is git-tracked, so worst case the deletion can
be reverted via `git checkout -- .beads/issues.jsonl`.)
