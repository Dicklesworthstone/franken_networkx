# franken_networkx Perf Baseline — 2026-05-03

> Generated by **cc_1** under `/profiling-software-performance` skill.
> Statistical comparison of fnx vs upstream NetworkX for the 5
> most-used graph algorithms across small / medium / large fixtures.

## Method

Algorithms (the 5 most-used in graph analysis):
1. `pagerank` — link analysis
2. `betweenness_centrality` — node importance via shortest-path counting
3. `dijkstra_path` — single-source-target shortest path with weights
4. `connected_components` — undirected component traversal
5. `clustering` — local clustering coefficient

Fixtures: `barabasi_albert_graph(n, 5, seed=42)` for `n ∈ {200, 1000,
5000}`. Same graph built independently in fnx and nx so node identity
and edge set are identical.

Timing: Python `time.perf_counter()` driven from the same process
(both libraries imported into one Python interpreter — no PyO3-→-nx
subprocess overhead, no cold-start noise). For small/medium each
measurement auto-tunes `n_calls` until each sample takes ≥ 70 ms,
then takes 7 samples and aggregates the middle 5 (drop fastest +
slowest). For large sizes where each call already exceeds 1 s, take
3–5 single-call samples and report the median; betweenness on BA5000
is one sample (each call ≈ 30–60 s × 7 samples × 2 libraries ≈ 14
min was prohibitive for routine baselining).

`speedup = nx_time / fnx_time` (>1× means fnx is faster).

Raw data: `/tmp/perf_baseline.log` + `/tmp/perf_large.log` +
`/tmp/_perf_large.json`. Bench scripts: `/tmp/perf_baseline.py` +
`/tmp/perf_large.py`.

## Findings — operations > 2× slower than upstream

The user's gate: file `[perf]` beads for any operation > 2× slower than
upstream.

### dijkstra_path — 5–10× slower at every size

| size | fnx | nx | slowdown |
|---|---:|---:|---:|
| BA200 | 869 µs | 161 µs | **5.4×** |
| BA1000 | 6.83 ms | 1.48 ms | **4.6×** |
| BA5000 | 53.4 ms | 5.13 ms | **10.4×** |

Slowdown gets *worse* as graph grows. Already filed as bead
**`br-r37-c1-644fx`** ([reality-check] add nx.paths.dijkstra_path).
Augment that bead with this new evidence (slowdown grows from 4.6× at
BA1000 to 10.4× at BA5000 — superlinear, suggests a quadratic factor
in the wrapper).

### clustering — 5–7× slower at every size

| size | fnx | nx | slowdown |
|---|---:|---:|---:|
| BA200 | 14.6 ms | 2.81 ms | **5.2×** |
| BA1000 | 120 ms | 19.9 ms | **6.0×** |
| BA5000 | 973 ms | 133 ms | **7.3×** |

Slowdown grows mildly with size, consistent with a per-node Python loop
over neighbour pairs. Already filed as bead **`br-r37-c1-wh0x0`**
([reality-check] add nx.clustering.clustering — bundles transitivity
/ average_clustering / generalized_degree). New evidence: confirms
the gap holds at BA5000.

### connected_components — 2–4.5× slower at every size

| size | fnx | nx | slowdown |
|---|---:|---:|---:|
| BA200 | 87.8 µs | 45.5 µs | **1.9×** ⚠ |
| BA1000 | 577 µs | 278 µs | **2.1×** |
| BA5000 | 8.60 ms | 1.88 ms | **4.6×** |

**New finding** (not in the reality-check audit since BA200 was below
the 2× gate). Slowdown grows from 1.9× at BA200 to 4.6× at BA5000 —
classic O(V) overhead in the result-set construction. nx returns an
iterator; fnx materializes a list of sets. **NEW bead filed — see
below.**

### pagerank — 3.3× slower at large size only

| size | fnx | nx | slowdown |
|---|---:|---:|---:|
| BA200 | 1.00 ms | 1.22 ms | 0.82× (fnx faster) |
| BA1000 | 5.43 ms | 4.05 ms | 1.34× |
| BA5000 | 69.9 ms | 21.1 ms | **3.31×** |

**New finding.** The `perf(pagerank)` fix in commit `a04084e0` made
fnx pagerank competitive on small graphs (1.22× *faster* on BA200),
but it doesn't scale: by BA5000 fnx is 3.3× slower than nx. Likely the
inner-loop overhead is per-edge HashMap lookups that are still in the
hot path even after the precomputation, or nx switched to a sparse-
matrix backend somewhere along the way. **NEW bead filed — see below.**

### betweenness_centrality — 1.84–4.05× *faster* (no action)

Solid Rust performance. Speedup ratio narrows from 4.05× at BA200 to
1.84× at BA5000 because nx's BFS-driven implementation scales
similarly to fnx's. No bead.

## Takeaways

1. **fnx wins on heavy-compute / Brandes-style algorithms** — betweenness
   (1.8–4.0×), eigenvector_centrality (3.7×), subgraph_centrality
   (1.7×), is_tree (17.6×). These are pure CPU loops where Rust's
   advantage is direct.
2. **fnx loses on graph-traversal-with-many-Python-results** —
   connected_components, dijkstra_path, clustering all show
   per-result overhead crossing the PyO3 boundary. The slowdown
   grows with output size (O(V) result for connected_components,
   per-node clustering coefficient, per-step path).
3. **pagerank scales worse than nx past |V|=1000** — likely because
   nx switches to scipy.sparse for large graphs while fnx stays on
   Python-side dict materialization. Worth investigating whether the
   PyDict-from-Vec<CentralityScore> conversion is the bottleneck.
4. **The next perf-improvement vector is "minimize the Vec<X> →
   PyList<PyDict<...>> conversion"** — the algorithms themselves are
   competitive in Rust, but materializing results into Python objects
   is dominated by per-element overhead. A `_raw` interface that
   returns numpy arrays could help where applicable (centralities,
   distances).
